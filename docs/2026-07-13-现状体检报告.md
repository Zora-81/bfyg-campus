## 三、部署可行性评估

你的架构是**长连接 Node 服务(Express + Socket.io + SQLite)**，不是静态页+云函数。

| 目标平台 | 可行性 | 说明 |
|----------|--------|------|
| CloudBase **云函数** | ❌ 不行 | 不支持长连接 WebSocket；无本地文件持久化；冷启动丢内存库 |
| CloudBase **云托管(容器)** | ✅ 可行 | 支持 WebSocket；需挂载持久卷 + 改 4 处(见下)；单容器运行 |
| 腾讯云轻量/任意 VPS + Docker | ✅ 最稳 | 直接用现成 `docker-compose.yml`（nginx+node 已配好），零改造 |

**结论**: 想上 CloudBase 就走**云托管(容器)**，别走云函数。最省事的是直接 Docker Compose 跑在轻量服务器——你这套配置已经齐了。

## 四、部署到 CloudBase 云托管必须改的 4 处

1. **修 SPA 回退 bug (#1)**: `index.js` 把 `staticDir` 提到模块作用域，生产模式也由 node 直接服务静态文件（去掉 `isDev` 对非 API 路由的限制）。
2. **挂载持久卷 (#2/#3)**: 在云托管控制台把 `/app/server/data.db` 和 `/app/server/uploads` 挂到持久存储(文件存储/CFS)，否则重启即丢。
3. **设环境变量**: `JWT_SECRET`(必填,勿用兜底值)、`NODE_ENV=production`、`PORT`(云托管分配端口)。
4. **静态服务策略二选一**:
   - A. 单容器内 node 同时服务静态+API+Socket（推荐，去掉 nginx 依赖，云托管自带网关/HTTPS）；
   - B. 前端丢 CloudBase 静态托管、后端云托管——此时需把 `API_BASE='/api'` 改成后端绝对域名，并收紧 CORS(对应 #7)。
